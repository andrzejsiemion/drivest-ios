# Implementation Plan: Cost Category Settings

**Branch**: `009-cost-category-settings` | **Date**: 2026-04-20 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/009-cost-category-settings/spec.md`

## Summary

Add a "Settings" item to the ··· ellipsis menu on all three tabs. Settings opens a form where users can toggle individual cost categories on/off. Disabled categories are hidden from the Add Cost category picker but never affect existing cost entries. Preferences are persisted in UserDefaults via a shared `@Observable` `CategorySettingsStore` injected into the SwiftUI environment.

## Technical Context

**Language/Version**: Swift 5.9+
**Primary Dependencies**: SwiftUI, SwiftData (unchanged), Observation framework
**Storage**: UserDefaults for category preferences (see research.md — intentionally not SwiftData)
**Testing**: XCTest
**Target Platform**: iOS 17.0+
**Project Type**: Mobile app (iOS)
**Performance Goals**: Preference reads/writes are synchronous and trivially fast (<1ms)
**Constraints**: Offline-only; no schema migration; all 7 categories enabled by default on first launch
**Scale/Scope**: 7 fixed categories, 3 tabs affected, 2 new files, 5 modified files

| I. Clean Code | ✅ PASS | Single-responsibility: `CategorySettingsStore` owns preferences; `SettingsView` owns UI; no cross-cutting logic |
| II. Simple UX | ✅ PASS | Settings reachable in 2 taps from any tab; toggle = immediate effect, no Save button |
| III. Responsive Design | ✅ PASS | SwiftUI `Form`+`Toggle` adapts to all sizes; Dark Mode automatic |
| IV. Minimal Dependencies | ✅ PASS | UserDefaults + SwiftUI only; zero new packages |
| iOS Platform Constraints | ✅ PASS | iOS 17+, SwiftUI, MVVM, no server dependency |
| Navigation depth | ✅ PASS | Settings is a sheet (not pushed), stays within 3-level limit |

**UserDefaults vs SwiftData**: Justified in research.md. Category preferences are settings (7 booleans, fixed keys), not relational app data. UserDefaults is the iOS standard for this use case.
